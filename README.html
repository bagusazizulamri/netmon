NetMon v1.0.0
Monitoring Aktif

NetMon

Sistem Pemantauan Jaringan Real-Time yang ditenagai oleh Python, Flask, Flask-SocketIO, dan SQLite. Dilengkapi dengan auto-discovery SNMP, pemetaan interaktif denah lantai (floor plan) visual, manajemen zona, sinkronisasi UniFi Controller, serta log audit yang lengkap.

SNMP v1/v2c Protokol Polling
Real-Time WebSockets (Socket.IO)
Interactive Drag & Drop Floor Plan

Fitur Utama

SNMP Auto-Discovery

Melakukan scanning rentang CIDR (subnet) untuk mengidentifikasi perangkat yang merespons SNMP secara otomatis dan memasukkannya ke database.

Polling Otomatis Berkala

Background worker berbasis APScheduler yang melakukan query ke perangkat berkala guna memperbarui status (Up, Down, Warning) secara real-time.

Interactive Floor Plan

Unggah denah lantai kustom dan posisikan ikon perangkat secara drag & drop. Perangkat akan berdenyut (pulse) warna merah/kuning saat ada peringatan.

UniFi Sync Integration

Menghubungkan langsung ke UniFi Controller untuk mengimpor seluruh perangkat (Gateway, Switch, Access Point) beserta tipe dan alamat MAC-nya sekali klik.

Alert & Access Logging

Menyimpan sejarah kegagalan perangkat, alert yang butuh acknowledgment, serta audit trail dari pengguna yang mengakses dashboard.

Zone Management

Mengelompokkan perangkat berdasarkan lokasi fisik, lantai, atau segmen VLAN dengan pewarnaan yang unik pada floor plan.

Persyaratan Sistem

Pastikan server Anda memenuhi persyaratan minimum berikut untuk menjalankan NetMon dengan lancar. Anda dapat mencentang daftar di bawah ini untuk memverifikasi kesiapan lingkungan Anda:

Python 3.9 atau lebih baru Aplikasi menggunakan fitur modern Python dan library typing. Disarankan Python 3.10 / 3.11.
Sistem Operasi Kompatibel Linux (Ubuntu 20.04+, Debian 11+, CentOS 8+) untuk pemasangan otomatis via systemd, atau Windows 10/11 (manual).
Port Jaringan Terbuka Port TCP 5000 (untuk dashboard web) dan Port UDP 161 (outbound untuk SNMP query ke perangkat target).
Akses Jaringan SNMP Perangkat target (Router, Switch, Server) harus dikonfigurasi untuk menerima request SNMP GET/WALK dari IP server NetMon.
Pustaka C compiler & Python Dev Header Dibutuhkan saat build dependensi Python seperti puresnmp / eventlet (terutama pada Linux server minimalis).
Catatan Penggunaan Windows

Di Windows, polling berbasis SNMP dan penemuan jaringan berjalan dengan normal, namun skrip installer otomatis tidak dapat dijalankan secara langsung. Gunakan petunjuk instalasi manual Windows.

Petunjuk Instalasi

Pilih tab di bawah ini yang sesuai dengan lingkungan sistem operasi server Anda:

Gunakan installer otomatis `install.sh` untuk melakukan konfigurasi menyeluruh termasuk pembuatan user systemd, virtual environment, dependencies, database, dan rule firewall.

Bash Shell 
# 1. Clone repositori NetMon ke server Anda
git clone https://github.com/yourorg/netmon.git
cd netmon

# 2. Jalankan skrip instalasi dengan hak akses root (sudo)
sudo bash install.sh
Apa saja yang dilakukan installer?

Menginstal paket sistem dependencies (`python3-venv`, `net-tools`, `iputils-ping`), membuat user khusus `netmon` (untuk keamanan proses), membangun database SQLite, menyusun file service systemd `netmon.service`, mendaftarkannya agar berjalan otomatis saat boot, dan membuka port firewall 5000.

Di lingkungan Windows, Anda perlu mengonfigurasi virtual environment secara mandiri melalui PowerShell:

PowerShell 
# 1. Buka PowerShell dan arahkan ke direktori project
cd C:\netmon

# 2. Buat virtual environment python
python -m venv venv

# 3. Aktifkan virtual environment
.\venv\Scripts\Activate.ps1

# 4. Install dependencies dari requirements.txt
pip install -r requirements.txt

# 5. Jalankan aplikasi NetMon
python app.py

4. Mekanisme Kerja UniFi Sync Integration

NetMon dilengkapi dengan modul integrasi UniFi Controller yang memungkinkan sinkronisasi data seluruh infrastruktur UniFi (AP, Switch, Gateway) secara otomatis ke dalam database NetMon. Berikut adalah alur proses bagaimana integrasi ini bekerja:

Alur Proses Sinkronisasi UniFi

Proses sinkronisasi dilakukan melalui interaksi antara Frontend, API Routes Flask, UniFi API Client, dan Database SQLite. Berikut adalah rincian teknis dari langkah-langkahnya:

  1. Inisiasi Request (HTTP POST): Sinkronisasi dimulai ketika pengguna memicu sinkronisasi (baik dari halaman Settings maupun Devices) via POST request ke endpoint /api/unifi/sync. Request ini mengirimkan credentials UniFi (host, port, username, password, site). Jika parameter tidak dikirimkan, sistem akan otomatis mengambil nilai fallback yang tersimpan di tabel settings database.
  2. Proses Autentikasi Dual-Path: Kelas UniFiClient (pada unifi_client.py) diinisialisasi menggunakan sesi HTTP (requests.Session) dengan mematikan peringatan sertifikat SSL. Sistem mencoba login dengan skema dual-path untuk mendukung controller lama (port 8443) maupun UniFi OS modern (port 443) dengan mencoba endpoint login /api/login dan /api/auth/login secara berurutan.
  3. Pengambilan Data & Pemetaan Tipe: Setelah berhasil login, klien mengirimkan GET request ke endpoint API UniFi /api/s/<site>/stat/device untuk mengambil data perangkat mentah. NetMon memetakan tipe perangkat UniFi (seperti ugw/uxg/udm menjadi router, usw menjadi switch, dan uap menjadi access_point) menggunakan kamus TYPE_MAP internal.
  4. Pembaruan Database Cerdas (Upsert): Data hasil parsing dikirim ke metode add_or_update_unifi_device di database.py:
    • Update: Jika perangkat dengan IP yang sama atau unifi_id yang sama ditemukan, data name, mac, vendor, model, dan unifi_id akan diperbarui di database.
    • Insert: Jika perangkat belum terdaftar, NetMon akan menambahkannya sebagai perangkat baru dengan zone_id = 1 (Default Zone), status awal diimpor dari UniFi, dan menyetel snmp_enabled = 0 secara default karena statusnya dipantau melalui controller.
  5. Broadcast Real-Time (WebSockets): Setelah proses integrasi selesai di backend, Flask akan menulis rincian aktivitas ke dalam access_logs untuk audit trail. Selanjutnya, Socket.IO memancarkan event unifi_sync_complete secara real-time ke semua klien web yang sedang membuka dashboard guna memperbarui UI secara langsung tanpa muat ulang halaman.

Flow Logika Kode

NetMon dibangun menggunakan arsitektur modular yang membagi tanggung jawab antara penanganan antarmuka web, penyimpanan data persisten SQLite, dan kueri SNMP asinkron.

Secara garis besar, interaksi data dalam NetMon berputar pada tiga alur proses utama:

1. Alur Polling SNMP Berkala APScheduler (di app.py) memicu background_poll() setiap X detik → Membaca daftar perangkat dari database.py → Memanggil thread snmp_worker.py untuk melakukan kueri GET → Hasil disimpan ke DB → Event WebSockets dikirim ke browser.
2. Alur Scanning Jaringan User klik "Scan" di browser → Request POST dikirim ke app.py → Spawns thread baru menjalankan snmp_worker.py:scan_network() → Iterasi subnet IP menggunakan ICMP/SNMP → Perangkat baru dimasukkan ke DB → Progress dipancarkan real-time via WebSockets.
3. Alur Sinkronisasi UniFi User klik "Sync UniFi" → POST /api/unifi/sync di app.py → Memanggil kelas UniFiClient (pada unifi_client.py) → Login dan ambil data JSON dari controller → Tipe perangkat dipetakan → Fungsi Upsert di database.py dijalankan.
Integrasi Asinkron & Real-Time

Dengan membagi tugas polling ke dalam thread terpisah dan menggunakan Socket.IO untuk pembaruan UI, NetMon tidak mengunci antarmuka web saat melakukan scanning atau polling yang memakan waktu.

app.py — Titik Masuk Utama & Router

Berkas ini bertindak sebagai orkestrator utama aplikasi yang menghubungkan antarmuka pengguna dengan backend logis.

  • Inisialisasi Server: Menginisialisasi modul Flask, Flask-SocketIO (dengan opsi async_mode='threading' agar berjalan di lingkungan thread python biasa), dan database manager SQLite.
  • Seksi Background Scheduler: Menggunakan APScheduler untuk mendaftarkan pekerjaan interval background_poll() yang mengeksekusi pemeriksaan perangkat secara berulang berdasarkan setelan interval di database.
  • API Endpoints Handler: Mengatur API RESTful untuk memanipulasi data perangkat, manajemen zona, merespons permintaan scan jaringan, serta mengunggah gambar denah lantai kustom ke direktori static/img.
  • WebSocket Events Linker: Menangani koneksi awal WebSocket, mengirimkan data inisialisasi (init_data) yang mencakup seluruh status jaringan saat browser baru dibuka, serta merespons event instan dari client seperti permintaan force-poll perangkat.

database.py — Data Persistence Engine

Bertanggung jawab atas seluruh siklus hidup penyimpanan data persisten aplikasi NetMon menggunakan SQLite.

  • Database Initialization (_init_db): Secara otomatis membuat tabel-tabel utama (devices, zones, alerts, access_logs, floorplans, settings, snmp_metrics) jika berkas netmon.db belum ada di sistem saat aplikasi dinyalakan.
  • Thread-Safe Connection Context (conn): Menggunakan python @contextmanager untuk membagi koneksi SQLite di lingkungan multithreading. Mengaktifkan PRAGMA journal_mode=WAL untuk mengizinkan operasi read/write simultan, serta melakukan otomatis commit jika kueri sukses dan rollback jika terjadi kegagalan data.
  • Integrasi UniFi Upsert (add_or_update_unifi_device): Mengimplementasikan logika cerdas di mana jika ada perangkat UniFi baru, data dimasukkan. Jika perangkat sudah terdaftar berdasarkan IP atau UUID Controller, kolom properti perangkat akan diperbarui demi mencegah duplikasi.
  • Metode Utilitas Alerting (acknowledge_alert): Mengubah status peringatan aktif menjadi terbaca (acknowledged) beserta perekaman waktu dan nama admin yang mengeksekusi aksi.

snmp_worker.py — SNMP Driver & Scanner

Berkas ini berisi logika tingkat rendah untuk berinteraksi dengan protokol SNMP menggunakan library puresnmp.

  • Low-Level Wrappers (_get & _walk): Membungkus instansiasi client puresnmp.PyWrapper dengan parameter community string, IP target, port, versi SNMP (v1/v2c), dan timeout guna menangani eksepsi kehilangan koneksi secara aman.
  • Device Polling Logic (poll_device): Mengambil nilai OID standar (Deskripsi, Uptime, Hostname). Jika perangkat yang sebelumnya 'down' merespons, status diubah menjadi 'up' dan alert pemulihan dipancarkan. Jika perangkat gagal merespons, status diubah menjadi 'down' dan record alert baru akan dicatat.
  • Interface Status Check (_check_ifaces): Melakukan SNMP WALK pada OID interface ifOperStatus (operational status) untuk memverifikasi jika ada port penting perangkat yang mati (Down/Warning). Jika ada port mati, status visual perangkat akan beralih ke warna kuning (Warning).
  • Network Scanner (scan_network): Memecah alamat CIDR menjadi deretan IP individual menggunakan modul ipaddress. Melakukan ping ICMP dasar lalu dilanjutkan dengan SNMP GET deskripsi sistem. Setiap perangkat baru yang ditemukan akan dikirim ke database dan dikabarkan ke UI via WebSocket progres.

unifi_client.py — UniFi API Integrator

Menyediakan jembatan API HTTPS untuk mengambil data topologi jaringan langsung dari UniFi SDN Controller.

  • Login Dual-Path (login): Mencoba autentikasi HTTP POST pada jalur /api/login (untuk controller klasik port 8443) atau /api/auth/login (untuk UniFi OS modern) guna memastikan kompatibilitas yang luas pada firmware controller yang berbeda.
  • Data Fetchers (get_devices & get_clients): Mengambil daftar perangkat keras (APs, Switches, Gateways) dan perangkat klien nirkabel/kabel yang terhubung ke jaringan UniFi.
  • Unified Parsing (_parse_device): Melakukan standardisasi skema data mentah UniFi. Menyelaraskan atribut alamat MAC, uptime, status (Up/Down), serta memetakan tipe perangkat UniFi (seperti gateway model 'udm' menjadi tipe 'router' NetMon) menggunakan kamus TYPE_MAP agar cocok dengan struktur tabel perangkat SQLite.

API & WebSockets

NetMon menyediakan REST API yang bersih untuk integrasi pihak ketiga, serta WebSocket server berbasis Socket.IO untuk pembaruan data dinamis secara real-time.

API Endpoints

GET /api/devices
Mendapatkan daftar lengkap perangkat jaringan beserta zona dan data polling terbarunya.
POST /api/devices
Menambahkan perangkat baru secara manual ke database. Membutuhkan parameter JSON: name, ip, dan type.
PUT /api/devices/<id>
Memperbarui informasi perangkat yang sudah ada berdasarkan ID yang dikirimkan.
DELETE /api/devices/<id>
Menghapus perangkat secara permanen dari sistem pemantauan dan database.
POST /api/scan
Memulai scanning jaringan asinkron. Payload JSON: network (CIDR) dan community string.
GET /api/scan/status
Memantau progress scanning subnet yang sedang berlangsung (persentase, IP aktif, jumlah perangkat ditemukan).
POST /api/unifi/sync
Memicu sinkronisasi manual secara instan dengan UniFi Controller.

Event WebSocket (Socket.IO)

Klien web mendengarkan event berikut untuk mengubah tampilan UI secara dinamis tanpa reload halaman:

Nama Event Arah (Server/Klien) Deskripsi Data
init_data Server → Klien Dikirim pertama kali saat koneksi terbuka. Membawa payload data lengkap perangkat, zona, floor plan, dan alerts.
device_status_update Server → Klien Broadcast saat status up/down/warning perangkat berubah setelah proses polling.
new_alert Server → Klien Mengirim informasi peringatan baru ketika ada perangkat yang mati (Down) secara tiba-tiba.
request_poll Klien → Server Klien meminta server untuk melakukan kueri SNMP instan ke suatu perangkat berdasarkan device_id.

Troubleshooting

Berikut adalah solusi atas kendala yang paling sering ditemui dalam penggunaan aplikasi NetMon:

Pemberitahuan Status Perangkat Terbaca "Unknown" atau Selalu "Down"
Pastikan Anda telah mengaktifkan agen SNMP di perangkat target dengan parameter community string yang sesuai. Lakukan uji coba ping dari shell server ke IP perangkat target guna memverifikasi konektivitas jaringan. Periksa juga apakah ada firewall lokal (seperti iptables atau ufw) atau access list (ACL) perangkat yang memblokir port UDP 161.
Sinkronisasi UniFi Controller Mengembalikan Pesan Error "SSL/TLS Failed"
Aplikasi NetMon secara default mengabaikan verifikasi sertifikat SSL (menyetel verify=False) guna mengakomodasi self-signed certificate UniFi lokal. Namun, pastikan alamat host yang diisi tidak menyertakan skema https:// (cukup tuliskan alamat IP atau hostname domain saja seperti 192.168.1.10). Pastikan port UniFi sudah benar (port default controller klasik adalah 8443, sedangkan UniFi OS menggunakan port default HTTPS 443).
Port 5000 Sudah Digunakan (Address Already in Use)
Jika port 5000 telah terpakai oleh aplikasi lain (misal di macOS AirPlay), Anda dapat mengganti port binding dengan menyetel environment variable PORT sebelum menjalankan start script, contoh: export PORT=5005 && ./start.sh atau ubah file unit systemd service NetMon.
Penggunaan CPU Server Sangat Tinggi
Hal ini biasanya terjadi jika Anda menambahkan terlalu banyak perangkat SNMP dan menyetel Poll Interval terlalu cepat (misal di bawah 15 detik). Masuk ke halaman Settings di dashboard dan naikkan interval polling menjadi 60 detik atau lebih guna memberikan ruang kerja thread backend.

Codebase Review & Security Notes

Review mendalam dari arsitektur kode saat ini untuk kepentingan audit keamanan dan panduan pengembangan bagi developer di masa mendatang.

Security Risk

Ketiadaan Otentikasi Bawaan (Authentication Bypass)

Aplikasi saat ini mengikat (bind) antarmuka dashboard ke alamat 0.0.0.0 tanpa dilengkapi login form/mekanisme keamanan internal. Rekomendasi: Pasang NetMon di belakang reverse proxy seperti Nginx, Apache, atau Caddy, lalu aktifkan Basic Auth atau integrasikan OAuth. Batasi akses IP server hanya untuk administrator jaringan.

Security Risk

SNMP v1/v2c Community Cleartext

Protokol SNMP v1 dan v2c yang didukung saat ini mentransmisikan community string (sandi otentikasi) dalam bentuk plain-text di jaringan tanpa enkripsi. Rekomendasi: Gunakan Access Control List (ACL) pada sisi router/switch untuk membatasi kueri SNMP hanya dari IP address server monitoring. Kurangi hak akses SNMP community menjadi Read-Only saja (jangan berikan hak Write).

Performance

Skalabilitas Threading pada polling SNMP

Proses kueri SNMP saat ini menggunakan threading.Thread bawaan Python untuk setiap perangkat. Untuk instalasi skala besar (>150 perangkat), hal ini berisiko memicu bottlenecks. Rekomendasi: Migrasikan penanganan request asinkron menggunakan AsyncIO atau gunakan executor pool (seperti ThreadPoolExecutor) untuk mendaur ulang pekerja thread dan membatasi persaingan resource CPU.

Architecture

Pemanfaatan SQLite WAL Mode

NetMon mengaktifkan PRAGMA journal_mode=WAL pada database SQLite. Ini adalah pendekatan arsitektur yang sangat tepat karena memungkinkan operasi baca (read) dari web Flask dan operasi tulis (write) dari background scheduler SNMP berjalan bersamaan secara konkuren tanpa terjadi database lock.
