NetMon

Dokumentasi Pengembangan & Arsitektur Sistem

Periode Laporan: 1 Juni 2026 – 1 Juli 2026

1. Ringkasan Eksekutif

NetMon adalah aplikasi pemantauan jaringan (Network Monitoring) terintegrasi yang menyatukan penangkapan data SNMP otomatis, visualisasi topologi interaktif (Floor Plan), manajemen zona, antarmuka terminal interaktif (TUI), serta sistem penyaringan peringatan palsu cerdas bertenaga AI.

Selama periode pengembangan dari 1 Juni 2026 hingga 1 Juli 2026, NetMon telah melalui serangkaian pembaruan arsitektural besar untuk meningkatkan performa konkurensi (khususnya pada lingkungan Windows/WSL), stabilitas penangkapan data SNMP, efisiensi rendering antarmuka di sisi klien, serta integrasi kecerdasan buatan (Artificial Intelligence) untuk meredam alarm palsu (false warnings).

3. Desain Arsitektur Sistem

3.1 Alur Logika Analisis AI False Warning

Pendeteksi alarm palsu bertenaga AI bekerja di thread latar belakang untuk meminimalkan beban rendering utama. Berikut adalah alur logikanya:

  1. Perangkat gagal di-poll (SNMP Timeout/Ping gagal) -> Trigger alert baru -> Simpan ke database via add_alert.
  2. Fungsi asinkron analyze_alert langsung dipicu:
    • Menjalankan Ping Ulang Proaktif secara mandiri ke host. Jika sukses, alert diklasifikasikan sebagai is_false_alarm = 1.
    • Memeriksa konkurensi: jika ada >= 2 perangkat lain yang gagal dalam waktu 15 detik terakhir, alert ditandai sebagai *WSL Network Congestion* (koneksi virtual WSL tersedak).
    • Memeriksa Flapping: jika ada >= 2 alert sejenis dalam 15 menit terakhir, ditandai sebagai *Koneksi Flapping/Jitter*.
  3. Jika Google Gemini API Key tersedia, prompt komprehensif dikirim ke Gemini 1.5 Flash untuk mengevaluasi data jaringan secara holistik.
  4. Jika status terkonfirmasi sebagai False Warning:
    • Update kolom is_false_alarm = 1 dan tulis alasan AI di tabel alerts.
    • Ubah kembali status perangkat di database menjadi up (Online).
    • Kirim event alert_update dan device_status_update melalui Socket.IO untuk mengupdate UI browser instan (menghapus banner merah, merubah ikon peta menjadi hijau/biru).

3.2 Struktur Kode Utama

  • app.py: Mengatur rute API REST, memancarkan event Socket.IO, dan mengelola scheduler background.
  • database.py: Berisi skema database SQLite, logika migrasi otomatis kolom tabel, pencatatan deret waktu, dan kunci penulisan konkuren.
  • snmp_worker.py: Berisi mesin polling SNMP walk asinkron, rollover counter, dan log pings.
  • ai_detector.py: Berisi modul pendeteksi alarm palsu AI (integrasi REST API Gemini via urllib & aturan heuristik).
  • unifi_client.py: Berisi handler API UniFi Controller dan parsing data station klien.

4. Masalah Jaringan WSL & Solusinya

Saat aplikasi NetMon dijalankan di dalam Windows Subsystem for Linux (WSL), paket data UDP/ICMP dibagikan melalui vEthernet Virtual Switch. Karakteristik virtual switch ini membatasi throughput konkurensi untuk paket UDP kecil (seperti query SNMP).

Isu:

Query SNMP yang dilakukan secara konkuren ke 10+ perangkat secara bersamaan di detik yang sama akan dibuang oleh virtual switch WSL, memicu peringatan offline palsu (perangkat dilaporkan mati padahal hanya paket pollingnya yang dibuang).

Solusi:

  1. Pembatasan Thread Pekerja (WSL Concurrency Capping): Sistem membaca /proc/version dan jika teridentifikasi berjalan di WSL, membatasi thread pekerja status check menjadi 3 dan kolektor traffic menjadi 2.
  2. Ekstensi Waktu Tunggu Ping (Ping Timeout Extension): Meningkatkan toleransi timeout ICMP Ping menjadi 2.0 detik.
  3. AI Verification Fallback: AI detector secara proaktif menjalankan ping tunggal yang terisolasi saat alert dipicu. Karena berjalan mandiri (tidak berbarengan dengan pemindaian massal), paket ping tidak dibuang oleh vEthernet, memungkinkan pemulihan instan status perangkat ke up di database.

5. Panduan Konfigurasi AI False Warning Detector

  1. Navigasi ke Google AI Studio dan buat API Key gratis untuk model Gemini.
  2. Buka halaman Settings di menu navigasi sidebar kiri NetMon.
  3. Masukkan API Key Anda pada kolom input Google Gemini API Key di panel *AI False Warning Detector*.
  4. Klik tombol Test Connection. Sistem akan mengetes koneksi ke server Gemini dan menampilkan respons sukses berwarna hijau.
  5. Klik tombol hijau Save All Settings di kanan atas untuk menyimpan konfigurasi secara permanen.

6. Rincian Detail Pengerjaan Teknis (Technical Work Details)

Halaman ini menjelaskan rincian berkas yang dimodifikasi, potongan kode kunci, dan solusi masalah teknis secara mendalam.

6.1 Optimasi Kinerja Floor Plan Canvas (Sisi Klien)

  • Berkas yang Dimodifikasi: templates/floorplan.html
  • Detail Teknis:
    • Sebelumnya, loop animasi dijalankan tanpa henti (loop() memanggil draw() dan mendaftar dirinya ke requestAnimationFrame). Pada denah gambar resolusi tinggi, browser harus merender jutaan piksel secara konstan yang memicu spiking penggunaan CPU hingga 80%.
    • Saya merancang bendera renderRequested untuk membatasi pemanggilan gambar ulang hanya ketika ada perubahan visual nyata (Zoom, Pan, Dragging, atau pembaruan status data).
    • Metode checkAnimationNeed() memindai status seluruh zona. Jika tidak ada perangkat yang berstatus down dan tidak sedang terjadi dragging, loop animasi dihentikan sepenuhnya.
    • Efek pendar peringatan merah yang menggunakan ctx.shadowBlur digantikan dengan riak radar vektor berlapis. Ini memindahkan kalkulasi blur dari CPU ke GPU (Hardware-Accelerated rendering), menurunkan CPU browser ke <1% dalam kondisi idle.

6.2 Sistem Pendeteksi Peringatan Palsu AI (AI False Warning Detector)

  • Berkas yang Dimodifikasi: ai_detector.py (Baru), snmp_worker.py, database.py, app.py, templates/settings.html, templates/dashboard.html
  • Detail Teknis:
    • Ping Ulang Asinkron: Saat pemicu alert offline mendeteksi kegagalan, ai_detector.py memanggil thread baru untuk ping terisolasi. Jika ping ini sukses, status langsung dipulihkan ke up.
    • REST Client Tanpa Dependensi: Menggunakan urllib.request bawaan untuk mengirim JSON payload ke endpoint model gemini-1.5-flash Google API, menghindari pembengkakan requirements.txt.
    • Korelasi Alert Spikes: Logika deteksi menghitung alert bersamaan dalam rentang waktu 15 detik terakhir. Jika ada lonjakan kegagalan multi-perangkat, heuristik menyimpulkan adanya kemacetan port/switch WSL virtual daripada kematian perangkat fisik.

6.3 Peningkatan TUI (Terminal User Interface) Interaktif

  • Berkas yang Dimodifikasi: tui.py, app.py, database.py
  • Detail Teknis:
    • Mengimplementasikan rendering grafis berbasis karakter ANSI pada terminal menggunakan pustaka bawaan curses.
    • Membuat grafik lalu lintas RX/TX real-time terpisah menggunakan karakter half-block Unicode (▄ dan ▀) untuk menampilkan data visual resolusi tinggi dalam kisi-kisi terminal teks.
    • Pemuatan data antrean dan penangkapan data berjalan pada thread latar belakang terpisah untuk mencegah hambatan freeze masukan navigasi keyboard (tombol panah, esc).

6.4 Penanganan Counter Rollover & Traffic Spikes

  • Berkas yang Dimodifikasi: snmp_worker.py, database.py
  • Detail Teknis:
    • Perangkat jaringan memiliki register counter byte data 32-bit (maksimal ~4.29 GB). Jika lalu lintas tinggi, counter ini akan cepat penuh dan kembali ke 0 (rollover).
    • Logika perhitungan lama melakukan pengurangan langsung yang menghasilkan angka negatif besar atau lonjakan data palsu (spike) hingga belasan Gbps.
    • Saya mengimplementasikan rumus rollover math: traffic = (max_counter - old_value) + new_value jika new_value < old_value.
    • Menambahkan pembersih database mandiri (database_self_healing) untuk membersihkan data spike historis di atas 100 Gbps agar skala grafik kembali normal.
